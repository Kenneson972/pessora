Design Cycle 1 — Section 3 : Les 3 Edge Functions

Architecture détaillée des fonctions Supabase à créer

3A — create-subscription-session

Entrée (body JSON)
email?: string // pré-rempli si connu
price_id?: string // default: env STRIPE_ORA_PLUS_PRICE_ID
Auth
✗ Pas d'auth requise
Stripe collecte l'email directement en checkout
Ce qu'elle fait
1. Valide price_id (ou utilise STRIPE_ORA_PLUS_PRICE_ID)
2. Crée stripe.checkout.sessions.create({ mode: 'subscription', ... })
3. success_url → /abonnement/succes?session_id={CHECKOUT_SESSION_ID}
4. cancel_url → /commande/annulee
5. Retourne { url }

3B — stripe-webhook

Sécurité : Vérifie Stripe-Signature avec STRIPE_WEBHOOK_SECRET — rejette si invalide (401). Pas d'Authorization header Supabase.
checkout.session.completed
• Récupère email depuis session

3C — verify-subscription-session

Entrée (query param)
session_id: string // depuis l'URL succès
Sortie
{ status: 'processed' | 'pending' }
// 'processed' = webhook déjà passé
Logique
1. Récupère la session Stripe via session_id
2. Vérifie si subscription existe déjà en DB avec ce stripe_subscription_id
3. Si oui → retourne { status: 'processed' } (webhook a déjà tout fait)
4. Si non → exécute la même logique que checkout.session.completed
   (fallback race condition webhook/success page)
5. Retourne { status: 'processed' }
Pourquoi ce fallback ? En production, le webhook arrive généralement avant que l'utilisateur atterrisse sur la page succès. Mais en dev / latence réseau, il peut arriver après — ce fallback garantit que l'Ora+ est activé même si le webhook est lent.
